# Plain English Reports

Use this skill to write or edit reports so they are clear, concise, accurate, and reader-focused.

## Core Standard

Plain English means a message written with the reader in mind, in the right tone, using clear and concise language. It does not mean childish, over-simple, casual, or less precise writing.

When editing, preserve the meaning. Improve the route the reader takes to that meaning.

Always write in British English (en-GB) when using this skill, unless the user explicitly asks for another variety of English.

## British English

Use UK spelling, vocabulary, punctuation, and tone throughout. Treat GOV.UK style as the default reference for clear public-facing writing, while adapting it sensibly for technical documentation and the user's house style.

Use British spelling patterns:

- -our: colour, behaviour, honour, flavour
- -re: centre, theatre, metre, litre
- -ise and -yse: organise, recognise, analyse
- -ence: defence, licence, offence
- other common forms: travelling, grey

Avoid American terms when a normal UK term is available:

- elevator -> lift
- sidewalk -> pavement
- zip code -> postcode
- cell phone -> mobile phone
- soccer -> football, unless the context needs 'soccer' to avoid ambiguity

Use 'programme' for planned work, public services, events, broadcasts, and courses. Use 'program' for computer software.

Prefer an understated, professional tone. Avoid over-excited or sales-led wording such as 'game-changing', 'revolutionary', 'transformative', or 'supercharge' unless quoting source text or matching an established brand voice.

Use single quotation marks for direct quotes and terms used as terms, with double quotation marks only for quotes within quotes. In code, JSON, Markdown syntax, or project-controlled formats, preserve the format's required quotation style.

Before returning final text, check for common American spellings and vocabulary, especially 'color', 'behavior', 'center', 'organize', 'analyze', 'license', 'defense', 'traveling', 'gray', 'elevator', and 'soccer'. Keep 'program' only when referring to computer software.

## Quick Workflow

1. Identify the reader and purpose.
2. Decide what the reader needs to know, what they already know, and what can be left out.
3. Put the most important information first.
4. Use a simple structure with clear headings.
5. Rewrite sentences for active verbs, direct address, everyday words, and fewer useless words.
6. Read the result as if speaking to the reader.
7. Remove wording that sounds generic, inflated, promotional, or AI-generated.
8. Check that the final text uses British English consistently.
9. Check that the final text is accurate, clear, concise, and readable.

## Editing Checklist

When rewriting text in plain English:

- Keep the average sentence length around 15 to 20 words.
- Use British English spelling, vocabulary, punctuation, and tone.
- Put one main idea in each sentence, with at most one closely related point.
- Break long sentences instead of packing them with commas and clauses.
- Prefer active verbs: 'We reviewed the files' instead of 'The files were reviewed'.
- Use passive verbs only when they are kinder, more natural, or the doer is unknown or irrelevant.
- Use 'I', 'we', and 'you' when they make responsibility and meaning clearer.
- Use everyday words the reader will understand.
- Explain necessary technical terms the first time they appear.
- Replace nominalisations with verbs: 'decided' instead of 'made a decision'.
- Replace business-speak with ordinary words, especially when it hides a simple action.
- Cut padding, repeated ideas, and phrases that add no meaning.
- Use positive framing where possible: say what the reader can or should do.
- Use lists to break up dense information.
- Make each list item follow logically from its introduction.
- Vary sentence rhythm so the text does not feel mechanical.
- Replace vague claims with specific facts, sources, decisions, or actions.

## Report Planning

Before writing a report, answer these questions:

- Why is this report being written?
- Who asked for it, and who will read it?
- What decision, action, or understanding should follow from it?
- What information belongs in the report?
- What information belongs in an appendix or can be left out?

If the purpose can be stated in one sentence, write that sentence before drafting the report.

Use either an outline or a mind map:

- Outline: gather facts and ideas quickly, group them, rank them by relevance, then add headings.
- Mind map: put the topic in the centre, branch out with main ideas, then group and order them.

## Report Structure

Use only the sections the report needs.

For long reports, use this order:

- Title or title page
- Contents list, if useful
- Abstract, for formal or research reports
- Introduction
- Summary and conclusions
- Recommendations
- Discussion
- Appendix, if needed

For short reports, use this order:

- Title
- Introduction
- Discussion
- Summary and conclusions
- Recommendations
- Appendix, if needed

The writing order can be different:

- Introduction
- Discussion
- Summary and conclusions
- Recommendations
- Abstract, if needed
- Title or title page
- Contents list, if needed
- Appendix, if needed

## Section Guidance

Title:
- Make it specific enough to show the subject and purpose.

Introduction:
- Keep it brief.
- Explain the topic, background, request, method, and sources when relevant.
- Move long methods or source lists to an appendix.

Discussion:
- Use headings and subheadings.
- Put the most important material first.
- Begin each paragraph with its main point, then add detail or explanation.
- Include charts and tables where readers need them, not automatically in an appendix.

Summary and conclusions:
- State the purpose, main findings, conclusions, and how you reached them.
- Keep conclusions brief.
- Explain what can be learned and what actions or options are best.

Recommendations:
- Say clearly what should happen next.
- Make recommendations specific, practical, and tied to the evidence.

Executive summary:
- Use when busy readers need the main points without reading the full report.
- Include the purpose, main findings, conclusions, and recommendations.
- Do not bury decisions or actions in the full report only.

Appendix:
- Use for supporting material needed only by readers studying the report in depth.
- Do not put essential evidence in an appendix just to shorten the main report.

## Active And Passive Verbs

Prefer active verbs because they usually make writing shorter, clearer, and more personal.

Active pattern:

```text
doer + verb + thing
The team reviewed the policy.
```

Passive pattern:

```text
thing + form of "to be" + past participle
The policy was reviewed by the team.
```

Passive clues include:

- be
- being
- am
- are
- is
- was
- were
- will be
- has been
- have been
- had been

Use a passive when:

- the active version sounds unnecessarily hostile;
- the doer is unknown;
- the doer does not matter; or
- the passive version simply reads better.

Aim for most verbs to be active.

## Nominalisations

Watch for nouns made from verbs. They often make writing heavy and abstract.

Rewrite them as verbs:

- 'make a decision' -> 'decide'
- 'carry out an investigation' -> 'investigate'
- 'give consideration to' -> 'consider'
- 'make reference to' -> 'refer to'
- 'provide an explanation' -> 'explain'
- 'undertake a review' -> 'review'

If a sentence contains 'of the', 'for the', 'in the', or 'by the' after an abstract noun, check whether a hidden verb would be clearer.

## Lists

Use lists when a sentence contains several conditions, steps, options, reasons, or examples.

For short, simple lists, prefer flowing prose with commas or semicolons if that reads more naturally. Use bullets when they make dense information easier to scan.

For inline prose lists, include a conjunction before the final item. Do not write 'logs, metrics, traces' when the sentence means 'logs, metrics and traces'. Missing 'and' or 'or' makes prose read like a slide deck or technical inventory rather than a finished sentence.

Use a serial comma only when it improves clarity, prevents a misread, or matches the surrounding house style. British English does not require a serial comma in every list, but the final conjunction is still required in normal prose.

Technical exceptions are allowed:

- Keep comma-only lists in code, arrays, tags, labels, CLI arguments, JSON, YAML, table schemas, filenames, import lists, and other machine-oriented syntax.
- Keep terse comma-only inventories only when they are clearly labels or catalogue entries rather than sentences.
- If a table cell reads as prose, apply the normal inline-list rule.

For separate complete points:

- Start each item with a capital letter.
- End each item with a full stop.

For a continuous sentence:

- introduce the list with text that flows into every item;
- start each item with a lower-case letter;
- use semicolons after each item;
- use 'and' or 'or' before the final item; and
- end the final item with a full stop.

Check every list item by reading it together with the introduction. If it does not form a grammatical sentence, rewrite the introduction or the item.

## Colons

Use a colon when the text after it explains, defines, expands, or lists the thing before it.

Before keeping a colon, check the relationship between the two sides:

- If the second half names what the first half refers to, a colon is usually fine: 'The deployment has two shapes: multi-product and Alita-only.'
- If the second half gives a reason, result, or benefit, do not use a colon as shorthand. Write the relationship directly with 'because', 'so', 'that gives us', or a new sentence.
- If the first half is an instruction and the second half is a benefit, split the sentence. For example, replace 'Prefer the platform's way: shared runbooks, shared monitoring and shared on-call knowledge' with 'Prefer the platform's way. That gives us shared runbooks, shared monitoring and shared on-call knowledge.'

Be especially careful with sentences that look like 'Do X: benefit one, benefit two and benefit three'. They often sound clipped because the list is not the object of the instruction; it is the reason for the instruction.

## Tone

Write as if speaking to the reader in a professional conversation.

- Be direct without being rude.
- Be friendly without becoming chatty.
- Be precise without showing off.
- Use positive framing when it helps the reader act.
- Do not hide responsibility behind official-sounding language.

It is acceptable to start a sentence with 'And', 'But', 'Because', 'So', 'Or', or 'However' if it improves flow and clarity. It is also acceptable to end a sentence with a preposition or split an infinitive when that sounds more natural.

## Natural, Human-Sounding Prose

Plain English should not sound like generic AI copy. During the final edit, remove these patterns:

- Significance inflation: 'stands as', 'serves as a testament to', 'pivotal', 'crucial', 'vital role', 'underscores', 'reflects broader', 'evolving landscape', 'unlocks', 'paves the way for', 'lays the foundation for', 'the premise that makes X possible', 'what makes this possible', 'the key to', 'the foundation of'. State the cause directly: 'X happens because of Y' rather than 'Y is the premise that makes X possible'.
- Promotional language: 'groundbreaking', 'renowned', 'vibrant', 'rich' used vaguely, 'showcasing', 'boasts', 'must-visit', 'stunning'.
- Vague attribution: 'experts say', 'industry reports suggest', 'observers note', unless the source is named or the claim is made specific.
- Superficial -ing chains: 'highlighting', 'ensuring', 'reflecting', 'contributing to', or 'showcasing' when they only decorate the sentence.
- Copula avoidance: use direct verbs such as 'is', 'are', and 'has' instead of inflated phrases like 'serves as', 'stands as', 'features', or 'boasts'.
- Metaphorical shorthand for benefits: 'compounds', 'pays dividends', 'moves the needle', 'force multiplier', 'snowballs', 'network effects' used vaguely. These sound insightful but hand-wave in place of naming the benefit. Replace with the specific outcome, or, if the concrete examples are already in the sentence, remove the metaphor and let the examples stand. For example, 'consistency with the platform compounds: shared runbooks and shared monitoring' becomes 'prefer the platform's way. That gives us shared runbooks and shared monitoring'.
- Formulaic contrasts: avoid overusing 'not only... but also' and 'it is not just about...'.
- Rhetorical repetition for emphasis: avoid parallel-clause triples borrowed from speechwriting, such as 'No X, no Y, no Z', 'Not yesterday, not today, not tomorrow', or 'No code, no tests, no ceremony'. These drive a point home in a speech but sound out of place in a report or proposal. State the underlying fact once in plain prose.
- Neat rules of three: avoid three-part lists constructed for rhythm rather than because there are exactly three relevant items. If the third item is filler, cut it; if the list could honestly be four or five items, use a proper bulleted list.
- Meta-question framing: avoid phrases such as 'the question is', 'the question of whether', or 'the question "do we need X?" is answered by Y' when the sentence can state the rule directly. These phrases make the prose sound like a discussion transcript. Replace 'The question "do we need the LGTM stack here?" is answered by the platform bootstrap' with 'The platform bootstrap decides whether the LGTM stack exists' or 'Alita's product chart does not decide whether the LGTM stack exists'.
- Generic endings: replace 'the future looks bright' or 'exciting times lie ahead' with a concrete next step or conclusion.
- Chatbot artefacts: remove 'Of course', 'Certainly', 'I hope this helps', 'let me know', and similar assistant-like framing from user-facing copy.
- Excessive hedging: replace 'could potentially possibly be argued' with the simplest accurate level of confidence.

Do not add personality for its own sake in formal reports. Add voice by being specific, direct, and honest about uncertainty where it matters.

Ask one final audit question before returning polished text: what still makes this sound generic or AI-generated?

## Headings

Headings attract AI fluff just like sentences do. A good heading tells the reader what is in the section, not how the author feels about it. Replace headings that sound promissory, breathless, or like chapter titles in a motivational book.

Common AI-sounding heading patterns and how to fix them:

- 'What unlocks this', 'The key insight', 'The secret ingredient', 'The game-changer' -> name the thing directly. For example, 'Why reuse is now possible', 'The core decision', 'Why this works'.
- 'The journey ahead', 'Looking forward', 'The road ahead', 'What's next' -> use a concrete heading that names the content. For example, 'Next steps', 'Open questions', 'Timeline'.
- 'Setting the stage', 'Understanding the landscape', 'The bigger picture', 'A brief overview' -> 'Background' or 'Context', if that is what the section actually contains.
- 'What this means for us', 'What this means for you', 'Why this matters' -> name the consequence. For example, 'Impact on the schedule', 'Cost implications', 'Effect on the on-call rota'.
- 'Deep dive', 'Under the hood', 'Behind the scenes' -> name the subject. For example, 'How the sandbox router works', 'Database schema'.
- 'Key takeaways', 'Final thoughts' -> 'Summary', 'Conclusions', or 'Recommendations'.

Check every heading against two questions:

- Would a reader skimming the contents page know what is in the section?
- Does the heading promise more drama than the section delivers?

If the answer to either is wrong, rewrite the heading as a plain noun phrase that names the subject of the section.

## Dashes

Em dashes ('—') are one of the strongest fingerprints of AI-generated writing. Large language models overproduce them, typically unspaced and in places where a comma, colon, or full stop would read more naturally. During the final edit, treat every dash in the draft as a candidate for removal.

In British English, prefer a spaced en dash (' – ') over an em dash for a parenthetical break, consistent with GOV.UK style. Reserve em dashes for cases where source text or a project-controlled style already uses them.

Before the final pass, scan for every dash in the draft and ask:

- Would a comma serve? 'The platform team runs the cluster, so Alita inherits it.' is almost always better than 'The platform team runs the cluster — so Alita inherits it.'
- Would a colon serve? Use a colon when what follows explains or lists what came before.
- Would a full stop serve? A dash often splices two full thoughts that belong in separate sentences.
- Would brackets serve? Use round brackets for an aside that the reader could skip.

Keep a dash only when:

- the break is genuinely sharper than a comma would carry;
- the sentence would lose its rhythm without it; or
- the surrounding text already uses dashes in the same role and consistency helps the reader.

Even then, use at most one dash pair per paragraph, and no more than a handful across a document. A page with a dash in every other sentence almost always reads better once most are replaced with commas, colons, or full stops.

Use an en dash without spaces for numeric and name ranges: 'pp. 10–15', '2024–2026', 'London–Edinburgh route'. Use a hyphen for compound modifiers: 'customer-facing', 'en-GB'. Do not use an em dash for any of these.

## Business-Speak

Avoid corporate or consultant-style wording when an ordinary word says the same thing. Do not use long words to make simple actions sound more important.

Replace business-speak such as:

- utilise -> use
- leverage -> use, build on, or make the most of
- facilitate -> help, enable, or run
- optimise -> improve, unless it means technical optimisation
- streamline -> simplify, shorten, or improve
- initiate -> start
- deliverables -> work, outputs, or tasks
- stakeholders -> people affected, users, teams, customers, or the named group
- going forward -> from now on, next, or delete it
- at scale -> widely, for many users, or delete it if vague
- best-in-class -> best, leading, or the specific quality meant
- drive value -> improve, save money, help users, or state the actual result
- unlock potential -> help, allow, or state the actual benefit
- low-hanging fruit -> quick wins or easiest tasks
- reach out -> contact, ask, or speak to

Keep a business term only when it is a recognised term of art for the reader. If it stays, make sure the sentence says what someone will actually do.

## Common Rewrites

Use these swaps as prompts, not rigid rules:

- additional -> extra
- advise -> tell
- applicant -> you
- commence -> start
- complete -> fill in
- comply with -> keep to
- consequently -> so
- ensure -> make sure
- forward -> send
- in accordance with -> under
- in excess of -> more than
- in respect of -> for
- in the event of -> if
- on receipt -> when we get or when you get
- on request -> if you ask
- particulars -> details
- per annum -> a year
- persons -> people
- prior to -> before
- purchase -> buy
- regarding -> about
- should you wish -> if you wish
- terminate -> end
- utilise -> use
- leverage -> use, build on, or make the most of
- whilst -> while

## Rewrite Pattern

When the user asks you to rewrite text:

1. Keep the substance and required legal, technical, or policy meaning.
2. Replace bureaucratic wording with plain wording.
3. Split long sentences.
4. Make the doer clear where possible.
5. Move secondary detail after the main point.
6. Use a list if it makes the information easier to scan.
7. Remove AI-sounding patterns, filler, and inflated claims.
8. Return the rewritten version first.
9. If useful, add a short note naming the biggest changes.

## Review Pattern

When the user asks you to review a report or draft, give concise feedback on:

- whether the purpose and audience are clear;
- whether the structure helps readers find what they need;
- where the main point is buried;
- sentences that are too long or overloaded;
- passive verbs that hide responsibility;
- jargon or technical terms that need replacing or explaining;
- nominalisations and padding;
- generic or AI-sounding phrasing; and
- missing conclusions, recommendations, or next steps.

Prioritise changes that help the reader understand or act.
